4. Implement `useProjectState()` hook:
   - **loadProjects**: `invoke<ProjectDto[]>('list_projects')`, sets `projects`, catches errors to `loadError`. Called in `useEffect` on mount.
   - **loadSkills**: `invoke<ManagedSkill[]>('get_managed_skills')`, sets `skills`. Called in `useEffect` on mount.
   - **selectProject(id)**: Sets `selectedProjectId`, then loads tools via `invoke<ProjectToolDto[]>('list_project_tools', { projectId: id })` and assignments via `invoke<ProjectSkillAssignmentDto[]>('list_project_skill_assignments', { projectId: id })`. Sets `matrixLoading` true before calls, false after. Use a version counter ref (`selectVersionRef`) to discard stale results: increment counter before fetch, check counter matches after fetch before setting state (per Pitfall 5 from RESEARCH.md).
   - **registerProject(path)**: `invoke<ProjectDto>('register_project', { path })`, then calls `loadProjects()`, returns the new ProjectDto.
   - **removeProject(id)**: `invoke('remove_project', { projectId: id })`, then calls `loadProjects()`. If removed project was selected, set `selectedProjectId` to null, clear `tools` and `assignments`.
   - **toggleAssignment(skillId, tool)**: Per D-06 wait-for-backend pattern. Add `"${skillId}:${tool}"` to `pendingCells` Set (use functional update: `prev => { const next = new Set(prev); next.add(key); return next }`). Check if assignment exists in current `assignments` array. If exists: `invoke('remove_project_skill_assignment', { projectId: selectedProjectId, skillId, tool })`. If not: `invoke('add_project_skill_assignment', { projectId: selectedProjectId, skillId, tool })`. In both success and catch: re-fetch assignments via `invoke<ProjectSkillAssignmentDto[]>('list_project_skill_assignments', { projectId: selectedProjectId })`. In finally: remove from `pendingCells`.
   - **bulkAssign(skillId)**: Per D-07. Get all tool keys from `tools` array. Set all `"${skillId}:${toolKey}"` as pending. Call `invoke<BulkAssignResultDto>('bulk_assign_skill', { projectId: selectedProjectId, skillId })`. Then re-fetch assignments. In finally: clear all pending keys for this skill.
   - **resyncProject**: `invoke<ResyncSummaryDto>('resync_project', { projectId: selectedProjectId })`. Re-fetch assignments after. **Return the ResyncSummaryDto** for toast display by callers.
   - **resyncAll**: `invoke<ResyncSummaryDto[]>('resync_all_projects')`. Re-fetch projects and (if selected) assignments. **Return the ResyncSummaryDto[]** for toast display by callers.
   - **loadToolStatus**: `invoke<ToolStatusDto>('get_tool_status')`, sets `toolStatus`.
   - **addTools(tools)**: For each tool string in the array, call `invoke('add_project_tool', { projectId: selectedProjectId, tool })` sequentially. Then re-fetch tools via `list_project_tools`.
   - **removeTools(tools)**: For each tool string, call `invoke('remove_project_tool', { projectId: selectedProjectId, tool })` sequentially. Then re-fetch tools and assignments (some assignments may have been cascade-deleted).
   - Modal setters: Simple `useState` setters for `showAddModal`, `showToolConfigModal`, `showRemoveModal`, `removeTargetId`.

5. Wrap all action functions in `useCallback` with appropriate dependencies. Use the pattern from App.tsx: `useCallback(async () => { ... }, [deps])`.

6. Error handling: All `invoke` calls wrapped in try/catch. Errors normalized with `err instanceof Error ? err.message : String(err)`. Errors from mutations are re-thrown so the caller (component) can show toasts. The `loadError` state is only for initial project list load failure.
