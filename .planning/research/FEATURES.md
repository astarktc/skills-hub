# Feature Landscape

**Domain:** Per-project plugin/skill distribution in a desktop tool management app
**Researched:** 2026-04-07
**Confidence:** HIGH (multiple real-world ecosystems analyzed, existing codebase well-understood)

## Ecosystem Context

Per-project plugin/skill distribution is a solved pattern across developer tooling. The core mental model is consistent everywhere:

1. A **central registry** holds all available plugins/skills/tools
2. **Project-level configuration** declares which subset applies to each project
3. A **resolution mechanism** makes the right subset available when working in that project
4. **Status feedback** tells the user what's active, missing, or stale

The specific implementations vary, but the user expectations are remarkably consistent. The patterns below are informed by: VS Code workspace extensions (`.vscode/extensions.json`), JetBrains required plugins (`.idea/externalDependencies.xml`), Claude Code's own skill system (`.claude/skills/`), Cargo workspaces, asdf `.tool-versions`, and devcontainer features.

Skills Hub occupies a unique niche: it is a _meta-distribution tool_ -- distributing skills TO per-project directories of OTHER tools. It does not define the per-project format (Claude Code, Cursor, etc. each have their own). It populates them.

## Table Stakes

Features users expect. Missing = product feels incomplete or broken.

| Feature                                      | Why Expected                                                                                                                                                                                                                                                                                                                                                                     | Complexity | Notes                                                                                                                                                                                                               |
| -------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Register a project directory                 | Every tool that does per-project anything starts with "tell me where the project is." VS Code opens folders, JetBrains opens projects, asdf looks at cwd. A picker-based registration is the absolute minimum.                                                                                                                                                                   | Low        | Tauri dialog.open already available. Validate path exists and is a directory. Default display name to directory basename.                                                                                           |
| Remove a registered project                  | Users need to undo registration. Every project manager has a remove/unregister action. JetBrains has "Remove from recent projects," VS Code has "Remove workspace."                                                                                                                                                                                                              | Low        | CASCADE delete on assignments. Must clean up deployed symlinks/copies in the project directory, not just DB records.                                                                                                |
| Assign skills to a project                   | The entire point of the feature. VS Code uses a JSON recommendations array, JetBrains uses checkboxes in UI. The checkbox matrix proposed in the design doc is the right interaction pattern.                                                                                                                                                                                    | Medium     | Per-tool granularity is necessary because Skills Hub targets 47+ tools. A skill useful for Claude Code may not be useful for Cursor in the same project.                                                            |
| Unassign skills from a project               | Bidirectional: if you can add, you must be able to remove. VS Code lets you remove from recommendations, JetBrains lets you remove required plugins.                                                                                                                                                                                                                             | Low        | Must physically remove the symlink/copy from the project directory, not just delete the DB record.                                                                                                                  |
| Immediate sync on assign/unassign            | When you toggle a checkbox, the symlink appears/disappears. This is VS Code's pattern: recommend an extension, it prompts immediately. JetBrains alerts immediately when a required plugin is missing. Users expect instant feedback, not a separate "deploy" step.                                                                                                              | Low        | Already designed as optimistic UI with immediate backend sync. The sync engine exists and is path-generic.                                                                                                          |
| Sync status indicators                       | Status must be visible per-assignment. Green/synced, red/missing, yellow/stale. Every tool management system shows status: VS Code extension badges (installed/not installed), JetBrains plugin status (enabled/disabled/needs update).                                                                                                                                          | Medium     | Four states: synced (green), stale (yellow for copy-mode), missing (red, symlink broken or removed), pending (gray, sync in progress). Status check needs to verify filesystem state, not just DB records.          |
| Sync Project button                          | Re-sync all assignments for one project. This is the "npm install" equivalent -- rebuild the expected state. Users need this after they manually delete files, after system updates, or after pulling from git.                                                                                                                                                                  | Low        | Iterate project assignments, call existing sync primitives for each. Return a report with per-assignment results.                                                                                                   |
| Sync All button                              | Re-sync every project at once. Cargo has `cargo build --workspace`, npm has `npm install --workspaces`. The "fix everything" button.                                                                                                                                                                                                                                             | Low        | Same as Sync Project but across all registered projects. Good for after updating a skill in the central library.                                                                                                    |
| Handle removed/renamed project directories   | If a project directory is moved or deleted, the app must not crash or show phantom entries. VS Code grays out missing workspace folders. JetBrains marks recent projects as "Path doesn't exist."                                                                                                                                                                                | Medium     | Detect on project list load (stat the directory). Show a warning badge. Offer to remove the stale registration. Do NOT silently auto-remove -- the directory might be on a disconnected network drive.              |
| Handle removed skills (orphaned assignments) | If a skill is deleted from the central library but has project assignments, those assignments are orphaned. JetBrains shows "Plugin not available" status.                                                                                                                                                                                                                       | Low        | Mark assignment status as "missing" (red). The CASCADE delete handles DB cleanup if the skill record is removed, but the symlink target is now broken. Offer cleanup action.                                        |
| User-configurable tool columns per project   | Most users use 2-3 AI tools, not 47. A 47-column matrix is unusable. The design doc correctly identifies this and proposes a `project_tools` table to let users choose which tools appear.                                                                                                                                                                                       | Medium     | Without this, the assignment matrix is a wall of checkboxes for 47 tools. This is the difference between a usable feature and a frustrating one. First-time setup: auto-detect installed tools and pre-select them. |
| Coexistence with global sync                 | The existing global sync must continue working. Adding per-project sync cannot break the original behavior. A skill can be synced globally AND to specific projects.                                                                                                                                                                                                             | Low        | These are independent code paths writing to different target directories. The key is that the design does not touch the existing sync_engine.rs at all -- just uses it with different target paths.                 |
| Search/filter in assignment matrix           | With 57+ skills, finding the right one needs search. VS Code has extension search, JetBrains has plugin search, every list UI with more than ~15 items needs filtering.                                                                                                                                                                                                          | Low        | Simple text filter on skill name/description. Already standard React pattern.                                                                                                                                       |
| .gitignore prompt on project registration    | Synced skill directories (`.claude/skills/`, `.cursor/skills/`) should typically be gitignored because they contain symlinks to machine-local paths. VS Code doesn't do this (extensions aren't in the project), but asdf's `.tool-versions` is the opposite -- it IS checked in. For Skills Hub, the deployed artifacts are machine-specific symlinks and should be gitignored. | Low        | Prompt-based, not automatic. Respect the user's git workflow. Some users may want to commit copies. Detection: check if the relevant directories are already in .gitignore.                                         |

## Differentiators

Features that set the product apart. Not expected, but valued when present.

| Feature                                                   | Value Proposition                                                                                                                                                                                                                   | Complexity | Notes                                                                                                                                                                                               |
| --------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Bulk-assign button ("All Tools")                          | One click to check all configured tools for a skill. The design doc specifies this. JetBrains has "Enable all" for plugin categories. Saves many clicks when a skill applies to every tool.                                         | Low        | Simple UI: button per row that toggles all tool checkboxes for that skill. Only toggles the user-configured tools, not all 47.                                                                      |
| Auto-detect installed tools for initial tool column setup | When registering a project, pre-populate the tool column selection based on which AI tools are detected as installed. The existing codebase already has tool detection (`get_tool_status`).                                         | Low        | Use existing `get_tool_status` to detect installed tools. Pre-check those in the tool column picker. User can override. This is the asdf pattern of auto-detecting available runtimes.              |
| Staleness detection for copy-mode targets                 | When sync mode is copy (not symlink), detect when the source skill has been updated but the deployed copy hasn't been refreshed. VS Code doesn't need this (extensions auto-update), but any copy-based deployment system needs it. | Medium     | Content hash comparison: hash stored at sync time vs current source hash. Symlink mode doesn't need this because changes propagate instantly. Mark stale assignments with yellow indicator.         |
| Skill grouping/presets ("Frontend Pack")                  | Define bundles like "Frontend Pack" (frontend-design + tailwind + shadcn) and assign the whole group to a project in one click. VS Code has extension packs, JetBrains has plugin groups.                                           | High       | Deferred in the design doc and correctly so. This is a v2 feature. The assignment matrix already allows multi-select, which covers 80% of the use case with less complexity.                        |
| Project templates                                         | "When I create a new Python project, auto-assign these 8 skills." VS Code has workspace templates, devcontainer has template repositories.                                                                                          | High       | Deferred and correctly so. Requires a template definition system, template application workflow, and template management UI. The manual assignment matrix is sufficient for v1.                     |
| Portable assignment manifests (YAML export/import)        | Export a project's skill assignments as a shareable YAML file. Import it on another machine. This is the `.tool-versions` or `devcontainer.json` pattern: declarative, portable, version-controllable.                              | Medium     | Deferred and correctly so. Machine-specific project paths make this tricky. The manifest would need to reference skills by name (not path) and resolve them on the target machine. Good v2 feature. |
| Clone assignments from another project                    | "Set up Project B with the same skills as Project A." JetBrains has "Copy settings from" when creating projects.                                                                                                                    | Low        | Straightforward: read assignments from source project, create matching assignments for target project, sync. Good polish feature but not essential for v1.                                          |
| Context-aware skill suggestions                           | Based on what tools/languages a project uses, suggest likely-relevant skills. JetBrains does this with "Suggested plugins based on project facets."                                                                                 | High       | Requires analyzing project contents (package.json, Cargo.toml, etc.) and mapping to skill relevance. Nice v2+ feature. The user knows their skills better than auto-detection for now.              |
| CLI companion for headless environments                   | `skillshub sync --project BDA` for SSH sessions and CI. Every mature tool has a CLI: VS Code has `code --install-extension`, cargo has its CLI-first workflow.                                                                      | High       | Deferred. The design doc correctly identifies this as a future enhancement. The desktop GUI is the primary interface for v1.                                                                        |

## Anti-Features

Features to explicitly NOT build. These are traps that look valuable but add complexity without proportional benefit, or actively harm the user experience.

| Anti-Feature                                   | Why Avoid                                                                                                                                                                                                                                                                                                           | What to Do Instead                                                                                                                                                                      |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Auto-sync / file watcher daemon                | Symlink mode already propagates source changes instantly -- a file watcher adds zero value for the primary sync mode. For copy mode, staleness detection + manual re-sync is simpler, more predictable, and uses zero background resources. No tool in the AI coding ecosystem has a file watcher for this purpose. | Rely on symlinks (instant propagation) + manual Sync buttons for copy mode. Staleness indicators tell the user when re-sync is needed.                                                  |
| Per-project skill versioning                   | The central library is the single source of truth. One version per skill, period. Per-project versioning adds a version resolution system, conflict handling, and upgrade workflow. It's npm's `node_modules` problem replicated unnecessarily.                                                                     | Central library = canonical. If a project needs a modified skill, install a separate skill with a different name.                                                                       |
| Fork/customize skills with upstream tracking   | Patch systems (like Nix overlays or git patches on top of upstream) are complex and fragile. If a user needs a modified skill, they should create a new skill.                                                                                                                                                      | Install a customized copy as a separate skill. The source_type/source_ref already track origin.                                                                                         |
| Automatic .gitignore modification              | Silently editing a user's git configuration is hostile. Even VS Code, which is deeply integrated with git, asks before modifying .gitignore.                                                                                                                                                                        | Prompt the user with a suggestion. Show the exact lines to add. Let them confirm or skip.                                                                                               |
| Cross-machine project sync                     | Project paths are machine-specific (`/home/alex/Projects/BDA` vs `/Users/alex/Projects/BDA`). Syncing the projects table across machines via dotfiles/chezmoi would require path remapping, conflict resolution, and platform-aware normalization.                                                                  | Each machine registers its own projects. Portable assignment manifests (deferred) solve the cross-machine use case more cleanly.                                                        |
| Nested project support (monorepo sub-projects) | A project directory should be a single registered entity. Supporting nested projects (workspace members inside a monorepo) adds hierarchy, inheritance, and scope resolution complexity. Cargo workspaces and npm workspaces took years to get right.                                                               | Register the monorepo root as the project. If sub-packages need different skills, register them as separate projects. Let the filesystem structure determine scope naturally.           |
| Drag-and-drop skill ordering/priority          | Skills are not prioritized -- they're either assigned or not. Ordering adds complexity (sortable lists, persistence, sync ordering) with no functional benefit. AI tools load all skills in a directory; order is irrelevant.                                                                                       | Skills are an unordered set per project per tool. The matrix UI reflects this correctly.                                                                                                |
| Real-time collaboration / team sync            | Skills Hub is a local desktop app. Adding real-time sync between team members requires a server, conflict resolution, and auth. This is a different product.                                                                                                                                                        | Portable assignment manifests (deferred) and version-controlled `.claude/skills/` directories handle team sharing via git, which is the natural collaboration mechanism for developers. |
| Undo/redo for assignment changes               | Assignment changes are immediately synced (create/remove symlink). Undo would require tracking history and reversing filesystem operations. The cost of a wrong assignment is trivially low -- just uncheck the box.                                                                                                | Immediate toggle is sufficient. No transaction log needed.                                                                                                                              |

## Feature Dependencies

```
Register project → Assign skills (requires a project to assign to)
Register project → User-configurable tool columns (requires a project to configure tools for)
User-configurable tool columns → Assignment matrix display (columns depend on tool selection)
Assign skills → Sync status indicators (nothing to show status for without assignments)
Assign skills → Sync Project button (nothing to sync without assignments)
Assign skills → Sync All button (nothing to sync without assignments)
Assign skills → Staleness detection (only meaningful after assignment + sync)
Content hash computation → Staleness detection (hash must exist to compare)
Tool adapter registry (existing) → User-configurable tool columns (adapter metadata drives column options)
Tool detection (existing) → Auto-detect installed tools for column setup
Central library (existing) → All assignment operations (source of truth for skill content)
Sync engine (existing) → All sync operations (path-generic primitives already work)
```

## MVP Recommendation

The design doc's three-phase approach maps well to a minimum viable feature set. Prioritize in this order:

### Phase 1: Must-ship (table stakes)

1. **Register/remove project directories** -- entry point for everything
2. **User-configurable tool columns per project** -- without this, the matrix is unusable with 47 tools
3. **Assign/unassign skills via checkbox matrix** -- the core interaction
4. **Immediate sync on toggle** -- instant feedback, no separate "deploy" step
5. **Sync status indicators** (synced/missing) -- user must see what's working
6. **Sync Project / Sync All buttons** -- recovery mechanism
7. **Coexistence with global sync** -- must not break existing behavior

### Phase 2: Essential polish

8. **Search/filter in assignment matrix** -- necessary at 57+ skills
9. **Handle removed/renamed project directories** -- graceful degradation
10. **Handle orphaned assignments** -- data integrity
11. **.gitignore prompt** -- prevents common git mistakes
12. **Auto-detect installed tools for initial column setup** -- smooth onboarding

### Phase 3: Differentiators (if time allows)

13. **Bulk-assign "All Tools" button** -- convenience
14. **Staleness detection for copy mode** -- completeness
15. **Clone assignments from another project** -- power user feature

**Defer to future milestones:** Skill grouping/presets, project templates, portable manifests, CLI companion, context-aware suggestions.
