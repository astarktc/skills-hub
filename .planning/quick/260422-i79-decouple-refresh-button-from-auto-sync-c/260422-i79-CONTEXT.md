# Quick Task 260422-i79: Decouple refresh button from auto-sync checkbox - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Task Boundary

The refresh button in My Skills page (FilterBar) currently only calls `loadManagedSkills()` which re-fetches skill data from SQLite — it does NOT re-download skills from their source repos. Additionally, `handleSyncAllManagedToTools` has a `if (!autoSyncEnabled) return` guard that prevents any sync when auto-sync is off.

The goal is to make the refresh button actually re-download/update all managed skills from their source repositories into the central library (~/.skillshub), and decouple this from the auto-sync checkbox. The auto-sync checkbox should only control whether syncs to global tool directories happen.

</domain>

<decisions>
## Implementation Decisions

### Refresh Behavior

- Re-download + conditional sync: For each managed skill, pull latest from its source repo into the central library. Then, if auto-sync is ON, also re-sync to tool directories. If auto-sync is OFF, only the library gets updated.

### Loading/Progress UX

- Use the existing loading overlay with per-skill progress messages (e.g., "Updating skill 3/12: my-skill"). Consistent with existing UX patterns like sync-all-to-tools.

### Error Handling Scope

- Continue updating remaining skills on failure, collect all errors, show a summary toast at the end. Same pattern as `handleSyncAllManagedToTools`.

### Claude's Discretion

- None — all areas discussed.

</decisions>

<specifics>
## Specific Ideas

- Current `handleRefresh` at App.tsx:937 just calls `loadManagedSkills()` — needs to be replaced with a proper update-all flow
- Backend already has `update_managed_skill_from_source` command that re-pulls from git — use this per skill
- The `handleSyncAllManagedToTools` function (App.tsx:1707) has the `if (!autoSyncEnabled) return` guard — the new refresh should call this conditionally after updates complete
- Existing loading overlay and action message pattern should be reused

</specifics>
